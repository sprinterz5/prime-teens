# PrimeTeens — Платформа сопровождения поступления. План проекта

> Версия 2 (2026-07-04) — **полная замена предыдущего плана** (портфолио-платформа для
> подростков). Новое направление: защищённая CRM + веб/моб. платформа для агентства,
> сопровождающего школьников и студентов при поступлении в зарубежные и местные вузы.
> Код пока не пишем — это карта, по которой будем строить.

---

## 0. Цель проекта

Создать защищённую веб- и мобильную платформу для сопровождения школьников и студентов
при поступлении в зарубежные и местные вузы. Система объединяет:

- CRM для лидов, профориентации и продаж;
- кабинеты родителей, студентов и кураторов;
- интеграции с Documentolog (ЭЦП), Kaspi Pay, Freedom Pay;
- поддержку трёх языков (RU / KZ / EN);
- модули аналитики и AI-ассистентов.

## 2. Основные модули

### 2.1 CRM и лидогенерация
- Интеграция с Instagram, Telegram Ads, формами сайта.
- Платный профориентационный ассессмент через Kaspi Pay / Freedom Pay.
- Отчёт по ассессменту и персональные рекомендации.
- Воронка: **Лид → Оплата → Ассессмент → Предложение → Договор**.

### 2.2 Договоры и ЭЦП
- Автогенерация договора (RU / KZ).
- Подписание через Documentolog API с ЭЦП (EDS).
- Автоматическая активация договора после подписания и оплаты.

### 2.3 Платежи
- Kaspi Pay API — моментальные переводы и QR-счета.
- Freedom Pay / Halyk Pay — карточные и рассрочные платежи.
- Учёт валют (KZT/USD), автоматический курс ЦБ.

### 2.4 Коммуникации
- Интеграции с WhatsApp Business, Telegram Bot API, Email (Mail.kz / Yandex 360).
- Централизованный чат по каждому ученику.
- Автоматические уведомления (SMS / Telegram).

### 2.5 Кабинеты и отчётность
- Родительский кабинет — задачи, оплата, файлы, прогресс.
- Кабинет куратора — задачи, материалы, отчёты.
- Студенческий кабинет — задания, дедлайны, материалы.
- Единый календарь (Google / Microsoft интеграция).

### 2.6 AI-ассистенты

| Назначение | Пример |
|---|---|
| Сводка по общению с родителем | «Папа обеспокоен IELTS, ждёт ответа по лагерю» |
| Анализ тональности | Отмечает раздражённые или рискованные диалоги |
| Автогенерация писем | Вежливые follow-up'ы на RU/KZ |
| Краткий отчёт о прогрессе ученика | Автоматический дайджест за неделю |

## 3. Безопасность и соответствие закону

- Хранение данных внутри Казахстана (Astana Hub / Beeline DC).
- Шифрование AES-256, резервное копирование.
- ЭЦП, хэширование действий (аудит Documentolog).
- Авторизация через Keycloak + SMS OTP (KazInfoCom).
- Полное соответствие Закону РК «О персональных данных».
- Отдельное условие: часть студентов — несовершеннолетние → нужно родительское
  согласие на обработку ПДн и на заключение/подписание договора от их имени.

### 4.1 Риск-заметка по стеку (снять до старта кода)

Спека выше — целевая enterprise-архитектура для команды из ~5 человек на 7–8 месяцев.
Если реализация идёт силами меньшей команды + AI-инструментов, три вещи стоит явно
решить заранее, а не по ходу:

- **Keycloak vs проще.** Keycloak даёт SSO/MFA/SMS OTP из коробки, но это отдельный
  сервис для деплоя и поддержки. Если команда маленькая — можно начать с
  Auth.js/Supabase Auth + SMS OTP через отдельного провайдера (Mobizon/SMSC) и
  мигрировать на Keycloak, когда появится реальная нагрузка/требование enterprise SSO.
- **Kubernetes/Terraform/ELK vs managed.** Для MVP это операционно тяжело без DevOps в
  штате. Managed-хостинг (например, контейнер на VPS/Cloud.kz + managed Postgres +
  Sentry/Logtail для логов) закрывает те же требования по хранению данных в РК на старте,
  K8s — когда есть реальный трафик и DevOps-ресурс.
- **AI-слой.** Самостоятельный хостинг Llama 3 — это своя GPU-инфраструктура и MLOps.
  На старте дешевле и надёжнее использовать API хостируемых моделей (Claude/GPT) из
  простого serverless-слоя; переход на self-hosted — отдельная оптимизация под масштаб/
  требования по данным, не блокер MVP.

Это не отменяет спеку — это последовательность: закладываем архитектуру под неё,
но не обязаны разворачивать весь enterprise-стек в первый спринт.

### 4.2 Что требует отдельного human/legal ревью, а не просто кода

Три модуля, где ошибка стоит реальных денег или юридической силы, а не просто бага
на сайте — Kaspi Pay/Freedom Pay (боевые транзакции), Documentolog/ЭЦП (юридическая
сила договора), обработка ПДн несовершеннолетних (Закон РК). Для них: сначала sandbox/
тест-режим у всех трёх интеграторов, реальные деньги/подписи — только после проверки
юристом шаблонов договора и согласий, и после ручного смоук-теста полного цикла
оплаты/подписания на тестовых аккаунтах.

## 6. Модель данных (черновая схема)

```
User
  id, email, phone, passwordHash/oauthId, role(enum), status, locale, createdAt
  Роли: ADMIN, SALES, CURATOR, PARENT, STUDENT

ParentStudentLink
  parentUserId, studentUserId, confirmed

Lead
  id, source(instagram/telegram_ads/site_form), name, contact, status(enum),
  assignedSalesId, createdAt, consent(ПДн)

Assessment (профориентационный ассессмент)
  id, leadId/studentId, paymentId, status, reportUrl, recommendations{}, completedAt

Deal (сделка/воронка)
  id, leadId, salesId, stage(лид→оплата→ассессмент→предложение→договор), value, currency

Contract
  id, dealId, studentId, parentId, templateLocale(ru/kz), documentologDocId,
  signatureStatus, signedAt, fileUrl

Payment
  id, contractId/assessmentId, provider(kaspi/freedom/halyk), amount, currency,
  fxRateUsed, status, providerTxId, createdAt

CuratorAssignment
  curatorId, studentId, assignedAt, active

Task (задачи по студенту — куратор/студент видят)
  id, studentId, curatorId, title, description, dueDate, status, attachments[]

MessageThread / Message (централизованный чат по ученику)
  threadId, studentId, participants[], channel(internal/whatsapp/telegram),
  senderId, body, sentAt

Notification
  id, userId, channel(sms/telegram/email), template, status, sentAt

AiSummary (AI-ассистенты)
  id, threadId/studentId, type(sentiment/summary/digest/draft_reply),
  content, generatedAt, model

AuditLog
  id, actorId, action, entity, entityId, diff{}, createdAt
```

Все таблицы — `createdAt/updatedAt`, мягкое удаление, `locale` где нужен i18n.

## 8. Структура репозитория (набросок)

```
prime-teens/
├─ apps/
│  ├─ web/                      # Next.js — сайт (лидген) + все кабинеты
│  │  ├─ app/(public)/          # маркетинг, формы захвата лида
│  │  ├─ app/(auth)/            # вход, Keycloak/Auth callback
│  │  ├─ app/(parent)/
│  │  ├─ app/(student)/
│  │  ├─ app/(curator)/
│  │  └─ app/(admin)/           # админка, сделки, отчёты
│  ├─ api/                      # Java 21 + Spring Boot 3 — ядро CRM
│  │  ├─ leads/ deals/ contracts/ payments/ curators/
│  │  └─ integrations/          # Kaspi, Freedom, Documentolog, WhatsApp, Telegram
│  └─ ai/                       # Python FastAPI — сводки, тональность, digest
├─ infra/                       # Terraform, k8s-манифесты, CI/CD (по мере роста)
└─ docs/                        # эта папка планов, ADR по интеграциям
```

Монорепо с явным разделением `web` (Next.js) / `api` (Spring Boot) / `ai` (FastAPI) —
frontend не привязан жёстко к тому, что бэкенд на Java; можно независимо ускорять/
менять части.

| **Итого** | ≈ 7–8 месяцев, MVP ≈ 3–4 месяца | |

### 9.1 Детализация Этапа 1 (MVP)

1. Домен, окружения (dev/staging/prod), Keycloak (или временная замена) с ролями.
2. Модель данных Lead/Deal/Contract/Payment (§6), миграции.
3. Формы захвата лида на сайте → `Lead` → воронка в CRM (кабинет продаж).
4. Интеграция Kaspi Pay + Freedom Pay в **sandbox**, приём оплаты за ассессмент.
5. Генерация договора (RU/KZ шаблоны) → Documentolog API в **тест-режиме** ЭЦП.
6. Ручной e2e smoke-тест полного цикла: лид → оплата → ассессмент → договор → подпись.
7. Юридическое ревью шаблонов договора и форм согласия на ПДн — **до** перехода в прод.
8. Переключение Kaspi/Freedom/Documentolog в боевой режим.
